PHASE 2 — SERVER VALUE BRIDGE PATTERN
Apply this pattern FIRST before all other transformations
═══════════════════════════════════════════════════════════════════════

This is the foundation of Option B. Every server-side value that was
previously injected into inline scripts must be moved to data- attributes
on a container element in the markup. The external JS then reads from there.

RULE: No server value may ever be written into a <script> block.
      All server → client data transfer happens through data- attributes only.

FILE PLACEMENT RULE:
      When a .aspx page already has a corresponding .js file anywhere
      in the existing project, add all new JS code to that existing file.
      When no .js file exists for a page, create the new .js file in the
      same folder as the .aspx file and mirror the existing naming
      convention used elsewhere in the project.
      Never move, rename, or reorganise any existing file or folder.

  ── SHARED WEBMETHOD FETCH HELPER ──

  Add this helper function to your shared utils .js file
  (or at the top of each paired .js file if no shared file exists).
  This is the single function all WebMethod calls go through:

    /**
     * Call an ASP.NET WebMethod on the current page.
     * @param {string} methodName  - Exact method name as defined in .cs
     * @param {object} params      - Parameter object matching method signature
     * @param {string} pageUrl     - Page URL from data-page-url on page-config
     * @returns {Promise<object>}  - Parsed d property from WebMethod response
     */
    function callWebMethod(methodName, params, pageUrl) {
        return fetch(pageUrl + '/' + methodName, {
            method: 'POST',
            headers: {
                'Content-Type': 'application/json; charset=utf-8'
            },
            body: JSON.stringify(params)
        })
        .then(function (response) {
            if (!response.ok) {
                throw new Error('HTTP ' + response.status);
            }
            return response.json();
        })
        .then(function (data) {
            // ASP.NET WebMethod wraps the return value in a "d" property
            // This is a security feature — always unwrap it here
            return data.d;
        })
        .catch(function (err) {
            console.error('WebMethod call failed [' + methodName + ']:', err);
            throw err;
        });
    }

── OPTION 4B: HttpModule (use only if per-path variation is needed) ──

  Place CspHeaderModule.cs in the same folder as other HttpModule
  or App_Code files already in the project. Do not create new folders.

    public class CspHeaderModule : IHttpModule
    {
        private const string CspHeader =
            "default-src 'self'; " +
            "script-src 'self'; " +
            "style-src 'self'; " +
            "img-src 'self' data:; " +
            "font-src 'self'; " +
            "connect-src 'self'; " +
            "frame-ancestors 'none'; " +
            "form-action 'self'; " +
            "base-uri 'self';";

        public void Init(HttpApplication app)
        {
            app.PreSendRequestHeaders += OnPreSendRequestHeaders;
        }

        private void OnPreSendRequestHeaders(object sender, EventArgs e)
        {
            var app = (HttpApplication)sender;
            app.Response.Headers.Set("Content-Security-Policy", CspHeader);
        }

        public void Dispose() { }
    }

  Register in Web.config:
    <system.webServer>
      <modules>
        <add name="CspHeaderModule" type="CspHeaderModule" />
      </modules>
    </system.webServer>
